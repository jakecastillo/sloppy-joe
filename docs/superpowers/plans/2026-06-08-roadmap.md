# Sloppy Joe — Roadmap (2026-06-08)

From a 5-lens, repo-grounded planning pass (product/wedge · tech-debt · capability · DX/OSS · testing/release). The reviewers ran the code; the headline finding is that the three things that *are* the product were broken or dead while the surrounding engineering was strong.

## Recommended immediate
**Fix the flagship demo so `sloppy inject` actually fires and writes a verifiable audit chain.** It's the literal first-user moment *and* a hard blocker on the #1 strategic move (Phase-0 demand validation): you can't ask design partners "is your duct tape painful enough to replace this?" while the first command they run is a visible no-op with an empty audit log.

## NOW (active backlog — see BACKLOG.md)
1. **Fix flagship demo** — `sloppy inject --now` + a no-`for:` demo rule + README reconcile. *(S, decisive; no deps — first.)*
2. **`sloppy rules validate`** — zero-infra CI first-win; embodies "policy is a reviewed file in Git." *(S, independent.)*
3. **Enforce `intent_budget` + `rollback: on_clear`** — turn parsed-but-dead governance into the real product. *(M.)*
4. **Thread `ctx` through `Store` + bound Redis keys (TTL)** — do before the ledger move (changes every signature). *(S/M.)*
5. **Persist + bound the cost ledger behind `state.Store`** — survive restarts; stop the unbounded leak. *(M, after #4.)*
6. **Phase-0 demand-validation kit (docs)** — kill-criteria table, discovery script, target profile, demo script. *(M; human follow-up to recruit + record.)*

## LATER (demand-gated)
Richer AI-native signals (`eval.quality_regression`, `guardrail.tripped`) · approval gates over `ee/` RBAC · Prometheus `/metrics` · Store consistency docs · real-Redis integration job. **Parked YAGNI until a design partner pulls them:** OTLP traces, Postgres backend, Sigstore keyless signing, xDS/CRD Envoy driver.

## Sequencing
Demo fix first (unblocks everything user-facing). Then `rules validate` and `intent_budget`/`rollback` in parallel. The `ctx` sweep precedes the ledger-persistence move and the Redis-TTL work (it changes every `Store` signature — reverse order re-touches every method). Phase-0 can start once the demo fires; record the asciinema only after `rules validate` exists. In NEXT: LiteLLM correctness gates both the compose demo and the integration test; `v0.1.0` is tagged last so the first public tag captures a working, honest artifact.
